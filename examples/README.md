# Examples

Synthetic, schema-valid example artifacts demonstrating IncidentIQ's data shapes end to end. Every
file here uses invented company/service/incident names -- no real production data of any kind.

## Files

| File | What it is | Provenance |
| --- | --- | --- |
| `ecommerce-checkout-input.json` | A full `Incident` object (the `sample-ecommerce-checkout` bundled scenario), including all 14 evidence items, **before** any AI analysis has run. | Bundled sample data (`server/src/data/incidents/ecommerceCheckout.ts`), not AI-generated. |
| `database-connection-leak-input.json` | A full `Incident` object (the `sample-db-connection-leak` bundled scenario) before analysis. | Bundled sample data, not AI-generated. |
| `example-analysis-output.json` | A complete, schema-valid `AnalysisRun` produced by analyzing `sample-ecommerce-checkout`. | **Mock provider output** (`MockAIProvider`, deterministic, offline) -- see the `_provenance` field at the top of the file. **This is not real OpenAI or Anthropic output.** |
| `example-skeptic-review-output.json` | A complete, schema-valid `SkepticReview` challenging the leading hypothesis of the analysis run above. | **Mock provider output.** Same caveat as above. |
| `example-postmortem.md` | A postmortem draft for `sample-ecommerce-checkout`, rendered as standalone Markdown exactly the way the app's "Download as Markdown" export produces it. | **Mock provider output**, rendered through the same `buildPostmortemMarkdown`-equivalent logic the app itself uses. A provenance comment is the first line of the file. |

## How these were produced

The `*-input.json` files are the actual bundled `Incident` objects, unmodified. The three output
files were generated by running the real production pipeline end to end (prompt building -> mock
provider `complete()` -> response validation -> the same `mapAiResponseToAnalysisRun`/
`mapAiResponseToSkepticReview`/postmortem-mapping functions the live application uses) -- they are
not hand-written or invented; they are genuine output of this codebase's own logic, using the mock
provider specifically because it requires no API key and is fully reproducible.

**No example here is presented as real-provider output**, because no real-provider output has been
saved into this repository. See `docs/ai-tools-and-apis.md` for the honest record of what real
OpenAI verification *has* been done (in a prior development session, documented but not saved as a
committed example file) and `docs/experiments/` for the critical-AI-experiment framework that can
produce a saved, real-provider comparison when run with a real API key.

## Loading these examples

- The two `*-input.json` files match the `Incident` shape (`shared/schemas/incident.schema.ts`) and
  can be parsed with `IncidentSchema.parse(JSON.parse(fs.readFileSync(...)))`.
- `example-analysis-output.json` and `example-skeptic-review-output.json` match `AnalysisRun`
  (`shared/schemas/analysisRun.schema.ts`) and `SkepticReview` (`shared/schemas/skepticReview.schema.ts`)
  respectively, aside from the leading `_provenance` field, which Zod's default (non-strict) parsing
  simply ignores as an unrecognized key -- no stripping needed before validating.
- `example-postmortem.md` is plain Markdown, readable on its own with no tooling.

## No secrets

None of these files contain an API key, an authorization header, or any other secret -- the incident
data is entirely synthetic, and the AI-generated content came from the mock provider, which never
makes an external call at all.
