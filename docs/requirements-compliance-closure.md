# Requirements Compliance Closure Report

**Purpose:** this document records, item by item, what changed since `docs/requirements-compliance-audit.md`
(the original, evidence-based audit) as a result of the compliance-closure work performed on branch
`feature/final-compliance-closure`. It does **not** overwrite or edit the original audit — that
document remains the historical record of the state it audited. Every status below was assigned only
after verifying the corresponding code change is implemented, wired into the running application, and
covered by passing automated tests (or, for pure-documentation items, that the required document
actually exists and covers what was asked) — never based on documentation claims alone.

**Status values used (exactly five, no others):** `RESOLVED`, `IMPROVED`, `DOCUMENTED LIMITATION`,
`STILL OPEN`, `NOT APPLICABLE`.

## An honest note on M32/M38/M39

These three items are the clearest example of this project's completion-integrity standard in
practice. The underlying code changes (a rewritten prompt, a new quality gate, a new targeted repair
pass) were fully implemented, wired into `analysisService.ts`, and covered by passing automated
tests well before any real-provider call was made — and a real OpenAI verification has now also been
performed (`docs/experiments/real-openai-v2-verification/`, 2026-07-24), confirming genuine,
measurable improvement over the original `v1` finding on all three items. They remain marked
`IMPROVED` rather than `RESOLVED` for a precise reason: the verification is a single real call
against a single scenario, coverage was incomplete even in that one run (2 of 3 hypotheses gained
contradicting evidence, not 3 of 3; 1 of several plausible reasoning-risk types was found, not all of
them), and LLM output is not deterministic, so this one successful-ish run does not establish that
the same result would recur reliably. Marking these `RESOLVED` on the strength of one real call would
overstate what was actually demonstrated — exactly the kind of overclaim this project's own rules
prohibit. Re-running the verification multiple times, or extending it across more of the six bundled
scenarios, would be the natural next step to build genuine statistical confidence before upgrading
these to `RESOLVED` — recorded as such in `docs/reflective-report.md`'s Future Improvements section.
